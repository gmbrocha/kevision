## "Mod 5" — provisional answer

The filename `mod_5_changelog.xlsx` mentions "Mod 5". `workspace_demo/` carries
a revision set labeled `"Revision #2 - Mod 5 grab bar supports"`, which
suggests **Mod / Modification** is a wrapper above revision sets — one Mod
bundles N revision sets that all get priced together for the owner. This is
still pending Kevin's confirmation (open question in `KEVIN_QUESTIONS.md`).

## File shape

- One sheet (`Sheet1`), 170 rows × 10 columns.
- Only **5 content rows** in the entire file. The remaining ~165 rows are
  intentional spacer rows so each entry block has vertical room for an
  embedded screenshot in the `Detail View` column.
- **3 embedded PNG crops** anchored to the spacer space next to each row group.
- Several merged cells inside `Scope Included` to stack numbered sub-items.

## Columns

| Letter | Header | Type | Notes |
|---|---|---|---|
| A | `Correlation ` | text/decimal | Stable per-sheet ID, format `<sheet_number_no_letters>.<sequence>` (e.g. `105.1`, `110.4`). Used as float in his cells but treat as label. |
| B | `Drawing #` | text | Sheet ID, e.g. `AD-105`, `AE-110 ` (trailing-space and dash style preserved). |
| C | `Revision # ` | text | Package label + date, e.g. `Revision #1 - 10/10/25`. |
| D | `Detail #` | text | One of: `N/A - Cloud Only`, `Cloud Only`, or `Detail <n>`. Distinguishes plan-level clouds from detail-level scope. |
| E | `Scope Included ` | text (often merged across rows) | Either a single description, or numbered sub-items `1)`, `2)`, `3)`, `4)` in stacked merged cells. |
| F | `Detail View ` | embedded PNG | Cropped screenshot of the cloud area on the source PDF. Sized to span ~14 rows × 2 cols. |
| G | *(no header)* | text | Ad-hoc symbology annotations he wrote for our benefit. **Do not reproduce in automated output.** |
| H | `Responsible Contractor` | empty | Downstream pricing-pass fill. Leave blank. |
| I | `Cost?` | empty | Downstream pricing-pass fill. Leave blank. |
| J | `Qoute Received?` | empty | Downstream pricing-pass fill. Leave blank. **Preserve the typo verbatim** until he renames it. |

## Row patterns observed

### Pattern 1: Plain cloud on a plan, no detail callout

```
A: 105.1
B: AD-105
C: Revision #1 - 10/10/25
D: N/A - Cloud Only
E: Remove Existing Masonry From Masonry Wall to enlarge existing opening as
   required for duct clearance coordinate opening size and location with
   mechanical
F: <embedded crop of the cloud on AD-105>
```

### Pattern 2: Cloud on a plan that *encloses* a detail callout

Two rows are emitted: one anchoring the plan-level cloud, one carrying the
scope from the referenced detail.

```
Row A:
  A: 110.4
  B: AE-110
  C: Revision #1 - 10/10/25
  D: Cloud Only
  E: Scope Included in Detail 4 - AE-110
  F: <embedded crop of the floor-plan cloud, showing the detail callout
     bubble inside it>

Row B:
  A: 110.4               (same correlation ID — same logical change)
  B: AE-110
  C: Revision #1 - 10/10/25
  D: Detail 4
  E (merged across multiple rows):
    1) Dash Lines Indicate exsiting abandoned msony opening, verify
       location of existing masonry openings. Reuse as practical for
       mechanical ducts. Infill unused openings with materials matching
       adjacent
    2) Electrical Equipment - See Electrical Drawings
       (note appears twice)
    3) Concrete Housekeeping Pad Coordinate with Electrical
    4) Rough Opening for Mech Duct, Coordinate with Mechanical for
       Locations and Size
  F: <embedded crop of AE-110 Detail 4 elevation drawing>
```

Notes:

- Both rows share the same `Correlation` ID (`110.4`). Treat it as a
  many-rows-per-logical-change key, not a primary key.
- `(note appears twice)` is his manual annotation that the same callout
  appears multiple times inside the detail, but he writes the scope only
  once. Counter to the "one row per duplicate" rule we previously assumed.
- Indices on numbered sub-items can have gaps in the merged-cell layout
  (rows 28, 35, 38, 41 in this file — there are spacer rows between).

## Mapping to `docs/pricing_deliverable_schema.md`

| Kevin column | Our schema |
|---|---|
| `Correlation` | new column on `pricing_changes`, `correlation_id` (text, format `<sheet>.<seq>`) |
| `Drawing #` | `pricing_changes.sheet_id` |
| `Revision #` | derived from `revision_sets.label` + `revision_sets.set_date` |
| `Detail #` | `pricing_changes.detail_ref` + new flag `cloud_only` |
| `Scope Included` (one bullet) | `pricing_changes.change_summary` |
| `Scope Included` (numbered list) | `pricing_change_lines.scope_text` rows |
| `Detail View` | new column `crop_image_path` (we already have these crops in the workspace) |
| `Responsible Contractor` / `Cost?` / `Qoute Received?` | new empty columns; pricing pass fills them |

The two-row "cloud + referenced detail" pattern needs:

- a self-reference on `pricing_changes`, e.g. `references_pricing_change_id`,
  so the cloud-only row can point at the detail row carrying the actual
  scope. Both rows share `correlation_id`.
