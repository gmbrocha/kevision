## Labeling Rules

Current rules live in:

- `CloudHammer/docs/labeling_workflow.md`

Summary:

- one class only: `cloud_motif`
- label all visible real cloud motifs in a crop
- leave revision triangles and revision digits unlabeled
- crops may contain multiple clouds; label each real cloud
- include bold, faint, thin, intersected, and partial clouds when real
- marker ROIs are context only, not target labels
- if a crop contains only ambiguous noise/non-cloud artifact, leave it unlabeled
- prefer consistent human review over preserving an API guess

Open labeling policy question:

- Some crops contain many small repetitive clouds marking the same copied/detail
  element. Decide later whether to keep labeling each distinct motif, merge
  nearby repeated motifs, or treat dense repetition as a training-balance case.
  Until decided, label each distinct visible real cloud motif and note dense
  repeat cases.

## GPT Prelabel Benchmark Idea

There is a strong candidate experiment before doing hundreds more labels:

Compare GPT raw labels against human-reviewed labels on images where human
review is known by timestamp.

Compare:

- GPT raw labels: `data/api_cloud_labels_unreviewed`
- human labels: `data/cloud_labels_reviewed`
- reviewed subset: files where reviewed mtime is newer than raw API mtime

Metrics to implement:

- image-level cloud/no-cloud accuracy
- box precision
- box recall
- IoU at `0.25`, `0.40`, `0.50`
- error rates by visual type/reason bucket
- GPT confidence calibration

Training experiments worth running:

- `human_only_204`
- `gpt_only_high_conf`
- `human_plus_gpt_high_conf`
- `gpt_pretrain_then_human_finetune`

Do not treat GPT labels as truth until this benchmark is done. The risk is
systematic bias: GPT may consistently overbox partials, miss faint clouds, or
label repeated geometry incorrectly.
